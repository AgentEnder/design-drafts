# Tinderbox Docs Portal — Brief

## Audience

Developers evaluating Tinderbox for their own repo. They're skimming. If they
don't see the value in 30 seconds, they close the tab.

There are two recognizable sub-types:

1. **Readers.** Want to understand *why* the tool exists before installing it.
   Will read the prose, skip the code, then read the code.
2. **Tinkerers.** Want to run the tool first and read about it second. Will
   ignore prose entirely if they can copy-paste a working example.

We don't know which group dominates. This draft is partly about finding out.

## Goals for this exploration

- Compare a **text-first** reading experience against a **playground-first**
  try-it-now layout. Which converts evaluators into installers faster?
- Compare a **single-column** prose layout against a **three-pane** reference
  layout. The reference page (`pages/api.html`) probably wants three-pane;
  the home page probably doesn't. Confirm or refute.
- Compare a **paper** theme (serif, off-white) against a **console** theme
  (mono, dark terminal-adjacent). The paper theme reads better; the console
  theme matches the product.

## Constraints

- No frameworks. The real site is hand-written HTML with a Markdown pipeline.
- Print-friendly. A surprising fraction of users print API references.
- Should look credible at v0.4 — not over-polished, not under-baked.
