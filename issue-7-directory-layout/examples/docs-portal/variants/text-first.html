Why short-lived branches need a TTL

Pick any active repository and run git branch -a | wc -l. The number is almost always larger than the number of people on the team, and it grows monotonically. A small fraction of those branches are alive — actively shipping — and the rest are sediment. Tinderbox is a tool for separating the two.

The premise is that a branch you have not touched in two weeks is, by default, dead. Maybe you'll come back to it; maybe you won't. Either way, leaving it on the remote indefinitely costs everyone a tiny amount of attention, and that cost compounds. The fix is to mark each branch with a deletion date when you create it. If you want to keep it past that date, you have to renew it. Otherwise, burn reaps it.

The reason this lives in a CLI rather than a git config option is that the metadata — when each branch was sparked, what its TTL is — has to live somewhere git itself doesn't track. Tinderbox stores that data in .tinderbox/branches.json at the repo root, alongside the lockfile and the editor config. It's a small file. You can read it.

A reasonable objection: doesn't this just push the problem one level down? Now instead of stale branches you have a stale branches.json. In practice, no — because list is fast and humans like running it, the file gets exercised. Stale entries surface within a day. Stale branches in raw git can sit for years.

If you want to try it, the quickstart takes about five minutes.
